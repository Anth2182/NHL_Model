What is this project?

The NHL Edge Finder is a sports betting edge calculator. You give it the sportsbook's odds for an NHL game, and it compares those odds against probabilities calculated by a statistical model built from historical game data. The difference between what the model thinks the probability is and what the sportsbook is implying — that's the edge.

A positive edge means the model thinks you're getting a better price than the game is actually worth. A negative edge means the sportsbook has priced that outcome more expensively than the model expects.

The Core Idea If your model says Team A wins 60% of the time, but the sportsbook's odds imply only 55%, you have a +5% edge on Team A. Over a large sample, consistently finding and betting positive edges is how sharp bettors make money.

Poisson Distribution

The foundation of the entire model is the Poisson distribution. It answers the question: "If a team is expected to score λ goals on average, what's the probability they score exactly k goals?"

P(k goals) = e−λ × λk / k!

For example, if a team's expected goals (λ) is 3.0:

Goals (k)Probability
05.0%
114.9%
222.4%
322.4%
416.8%
5+18.5%

The model calculates λ separately for the home team (lam_h) and away team (lam_a) based on each team's attack strength, the opponent's defensive strength, and other adjustments. 

def poisson_pmf(k, lam):
    return np.exp(-lam) * (lam ** k) / factorial(k)

Dixon-Coles Model

Pure Poisson has a flaw: it treats home and away goals as completely independent, but in real life low-scoring games (0-0, 1-0, 0-1, 1-1) happen slightly more or less often than Poisson predicts. Dixon-Coles adds a correction factor ρ (rho) to fix this.

dc_correction(gh, ga) = if 0-0: 1 − λ_h × λ_a × ρ if 1-0: 1 + λ_a × ρ if 0-1: 1 + λ_h × ρ if 1-1: 1 − ρ else: 1.0 (no correction needed)

ρ is typically a small negative number (e.g. -0.1), meaning 0-0 draws happen slightly less often than pure Poisson predicts in hockey — because teams tend to push for goals when tied.

Attack & Defense Ratings

The expected goals for the home team in a given game are calculated as:

λ_home = exp( log_μ + home_advantage + attack[home] − defense[away] ) λ_away = exp( log_μ + attack[away] − defense[home] )

Everything is on a log scale so that the parameters can be added/subtracted cleanly and then exponentiated to get a positive goal rate. Here's what each parameter means:

ParameterWhat it meansTypical value
log_μThe league-average log goals per game~1.1 (≈ 3 goals)
home_advBoost for playing at home~0.05–0.10
attack[team]How much better/worse than average at scoring−0.3 to +0.3
defense[team]How much better/worse than average at preventing goals−0.3 to +0.3

How the model is trained

The model finds the attack/defense values that make the observed scores most likely — this is called Maximum Likelihood Estimation (MLE). It uses scipy.optimize.minimize with L-BFGS-B to find the parameters that minimize the negative log-likelihood across all games.

Why log scale? Using log ratings means a team's attack and the opponent's defense are simply added/subtracted. Without the log, you'd need to multiply fractions, which is harder to optimize and can produce negative goal rates.

Time Weighting (xi)

A game from 3 years ago should matter less than a game from last week. The model applies exponential decay weighting to each game based on how many days ago it was played.

weight = e−ξ × days_ago

The parameter xi = 0.005 controls how fast old games decay in importance. With ξ = 0.005, a game from 200 days ago gets a weight of e^(-0.005 × 200) = 0.37 — about 37% as much influence as a game played today.

Score Matrix & Win Probabilities

Once we have lam_h and lam_a, the model builds a full score matrix — a 16×16 grid where each cell [gh][ga] holds the probability of the home team scoring gh goals and the away team scoring ga goals.

P(gh, ga) = Poisson(gh, λ_h) × Poisson(ga, λ_a) × dc_correction(gh, ga)

From this matrix, every outcome can be computed by simply summing the right cells:

OutcomeWhich cells
Home win (regulation)All cells where gh > ga
Away win (regulation)All cells where ga > gh
Draw / OTDiagonal cells where gh == ga
Home win (incl. OT)Home reg wins + 50% of draws
Home covers -1.5All cells where gh − ga ≥ 2
Over X.5 goalsAll cells where gh + ga > X.5
Why 50% of draws? When the model says "OT", it doesn't know who wins — hockey overtime is essentially a coin flip (actually slightly home-favored, but 50/50 is a reasonable approximation). So each draw cell contributes 50% to home win probability and 50% to away win probability.
Totals and the "push" For over/under bets, the model separates out push probability (exact total = the line) and calculates over_fair and under_fair by normalizing only over the non-push outcomes. This matches how sportsbooks settle totals.

Blending Prior + Current Season

One of the smartest parts of this model is how it handles the small sample size problem early in the season. After 5 games you don't have enough data to trust current season ratings, so the model blends in prior season data.

The final team rating is a weighted blend of three sources:

SourceWhat it isWeight (varies by games played)
Prior DC Full prior season Dixon-Coles ratings 80% early → 0% after 30 games
Current DC Current season Dixon-Coles ratings Grows from 0% → up to 40%
Current Metrics xG, high-danger chances, shots (current season) Heavy early, then ~60% after 40 games 
# Weight schedule based on current season games played
if   g < 10:  prior_w = 0.80
elif g < 20:  prior_w = 0.50
elif g < 30:  prior_w = 0.20
else:         prior_w = 0.00
Why this is clever Pure current-season models blow up in October because 5 games of data is noise, not signal. By leaning on last year's ratings early and gradually transitioning, the model stays stable all season long.

Current Season Metrics

Beyond win-loss ratings, the model tracks three underlying performance metrics per team and converts them into attack/defense adjustments using z-scores.

MetricWeightWhat it measures
xGoals For/Against per game50%Expected goals based on shot quality — the gold standard of possession metrics
High-Danger xGoals per game30%xG from the most dangerous scoring zones (slot, crease)
Shots on Goal per game20%Raw shot volume
attack_adjustment = 0.50×z(xGF/gp) + 0.30×z(HD_xGF/gp) + 0.20×z(SOG/gp) defense_adjustment = 0.50×z(xGA/gp) + 0.30×z(HD_xGA/gp) + 0.20×z(SOGA/gp)

These z-scores are then scaled by 0.10 before being blended in — keeping them from dominating the model when sample sizes are still small.

Why use xGoals instead of actual goals? Actual goal scoring has a lot of randomness (goalie hot streaks, lucky bounces, etc.). Expected goals strips that out and gives a cleaner picture of a team's underlying performance. Teams that consistently generate high-danger chances tend to regress toward those numbers over time.

Goalie Ratings (GSAx)

Goalies are adjusted using Goals Saved Above Expected (GSAx) — the most reliable goalie performance metric available. It measures how many goals a goalie prevented compared to what an average goalie would have allowed given the same shots.

GSAx = xGoals_faced − actual_goals_allowed GSAx_per_game = GSAx / games_played

A positive GSAx/G means the goalie is outperforming expectations. The model converts this to a log-scale adjustment and applies it to the opposing team's lambda:

log_adjustment = clip(GSAx_per_game, −0.40, +0.40) × 0.33 λ_home = exp( ... − away_goalie_log_adj ) ← good away goalie reduces home scoring λ_away = exp( ... − home_goalie_log_adj ) ← good home goalie reduces away scoring

The 0.33 scaling factor and ±0.40 clipping prevent extreme goalie ratings from overwhelming team-level ratings. It treats goalie impact as real but bounded.

Minimum games threshold Only goalies with 5+ games played are included in ratings. Below that, the sample is too small to distinguish skill from luck.

Back-to-Back Penalty

When a team played a game the previous day, they take a performance hit from fatigue. The model applies a flat penalty to their expected goals:

B2B_PENALTY = −0.04 (log scale) if team is on B2B: λ_team *= exp(−0.04) ≈ λ_team × 0.961

A 4% reduction in expected goals on a back-to-back. The app auto-detects B2B situations from the schedule file and pre-checks those boxes, but you can always override manually.

Why log scale for the penalty too? Same reason as the ratings — multiplying by exp(-0.04) on the lambda is equivalent to subtracting 0.04 on the log scale. It keeps all the math consistent.

American Odds → Implied Probability

American odds come in two flavors — negative (favorites) and positive (underdogs). Converting them to implied probability is straightforward:

Negative odds (e.g. −150): prob = 150 / (150 + 100) = 60.0% Positive odds (e.g. +130): prob = 100 / (130 + 100) = 43.5%

Notice that 60.0% + 43.5% = 103.5%. That extra 3.5% is the vig — the sportsbook's cut. 

def _american_to_prob(odds):
    odds = float(odds)
    if odds > 0:
        return 100 / (odds + 100)
    else:
        return abs(odds) / (abs(odds) + 100)

Removing the Vig

Before comparing model probabilities to market probabilities, the app removes the sportsbook's margin to get the fair (vig-free) probability. If it didn't do this, it would be comparing apples to oranges — the model outputs true probabilities, but raw implied odds include the book's cut.

raw_home = _american_to_prob(home_ml) → e.g. 0.600 raw_away = _american_to_prob(away_ml) → e.g. 0.435 total = 0.600 + 0.435 = 1.035 ← the vig fair_home = 0.600 / 1.035 = 57.97% fair_away = 0.435 / 1.035 = 42.03%

Now fair_home + fair_away = 100%. These are the probabilities the market is actually implying after stripping out the juice. 

def vig_free(ml_a, ml_b):
    a = _american_to_prob(ml_a); b = _american_to_prob(ml_b)
    tot = (a or 0) + (b or 0)
    return (a/tot if tot else None), (b/tot if tot else None)

Edge Calculation

Edge is simple once you have both numbers:

edge = model_probability − fair_market_probability

The app calculates edge for both home and away, then shows the side with the better edge.

EdgeColorInterpretation
≥ +5%■ Dark greenStrong edge — model strongly disagrees with the market
+2% to +5%■ Light greenModerate edge
0% to +2%■ Pale greenSlight edge — proceed with caution
−3% to 0%■ Light redSlight fade — market has slight edge over model
< −3%■ Dark redStrong fade — avoid this side
Edge ≠ guaranteed profit A +5% edge means you expect to profit 5 cents per dollar bet over a large sample. Any individual game can still go either way. The model is trying to identify long-run value, not predict single outcomes.

App Flow

  1. Date picker — defaults to today (Eastern Time). Pulls matching games from the schedule CSV.
  2. B2B detection — checks which teams also played yesterday and auto-flags them.
  3. Lines tab — enter home/away moneyline odds and select starting goalies.
  4. Totals tab — enter the O/U line and over/under odds.
  5. Run Predictions — model predicts expected goals → score matrix → probabilities → edge vs. market.
  6. Results — two tabs (Moneyline, Totals) sorted by edge, color-coded.

Future Improvements

🔌 Odds API Integration

Pull live odds automatically instead of typing them in. Eliminates manual entry and enables real-time scanning across all games.

📈 Kelly Criterion

Given an edge and your bankroll, calculate the optimal bet size using the Kelly formula instead of flat betting.

Puck line inputs removed for now but the model already calculates home_pl_prob. Easy to re-add once odds API is wired up.

Log bets you actually place and track realized edge vs. expected edge over time to validate the model.

Currently the schedule CSV needs manual updating. Could automate this with a daily pull from the NHL API.

🏒 Lineup Adjustments

Account for key forward/defenseman injuries. Right now only goalie quality is modeled — missing skater adjustments is a known gap.
